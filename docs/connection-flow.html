unity-cli 연결 구조 변경 요약

이번 변경은 사용자가 포트를 고르지 않게 하고, Unity 리로드나 listener 재시작 중에도 같은 프로젝트의 최신 연결 상태를 다시 확인해서 명령을 보내도록 정리한 것이다. 버전 검사는 기본적으로 엄격하지만, 필요하면 --ignore-version-mismatch로 명시적으로 생략할 수 있다.

이전 흐름

포트 중심

1. 대상 선택사용자가 --port를 직접 넣거나, CLI가 한 번 찾은 포트를 계속 사용했다.
↓
2. Unity 상태 변화컴파일, 도메인 리로드, listener 재시작 중 포트와 heartbeat 파일 상태가 어긋날 수 있었다.
↓
3. 명령 전송이미 잡아 둔 포트로 전송해서 끊긴 listener나 다른 상태를 볼 수 있었다.
↓
4. 실패 양상타임아웃, 잘못된 인스턴스 선택, 오래된 heartbeat 의존 문제가 생겼다.
→
현재 흐름

프로젝트 중심

1. 대상 선택--project, 현재 작업 디렉터리, 유일한 활성 인스턴스 순서로 프로젝트를 고른다.
↓
2. 최신 상태 재조회명령 전과 재시도마다 instance 파일을 다시 읽어 같은 프로젝트의 최신 listener를 찾는다.
↓
3. 즉시 health 확인/health로 listener가 준비됐는지, 프로젝트가 맞는지, PID와 timestamp가 유효한지 확인한다.
↓
4. 명령 전송기본은 health를 통과한 snapshot으로만 /command를 보낸다. --ignore-version-mismatch에서 구버전 connector의 health endpoint 미지원이 확인된 경우만 discovery 결과로 보낸다.

현재 명령 실행 파이프라인

DiscoverDiscoverInstance(project)
프로젝트 경로 기준으로 활성 instance를 찾는다. 여러 개가 애매하면 에러를 낸다.
HealthGET /health
Unity main thread command queue를 거치지 않고 listener 자체 상태를 빠르게 확인한다.
SendPOST /command
보내기 직전에도 다시 resolve해서 리로드/재시작 후 새 listener를 따라간다.
포트는 내부 HTTP transport에만 남아 있다. 사용자는 포트를 입력하지 않고, status/help/README/로그/CLI 연결 에러에서도 포트 번호를 보지 않는다.

버전 검사 정책

기본값

버전이 다르면 거절

CLI와 Unity Connector가 서로 다른 릴리스면 명령 실행 전에 멈춘다. 이 경로는 프로토콜이나 응답 shape가 달라졌을 때 잘못된 결과를 막기 위한 기본 안전장치다.

명시적 우회

--ignore-version-mismatch

사용자가 이 옵션을 주면 버전 검사를 생략한다. 새 connector면 health 확인을 쓰고, health endpoint가 없는 옛 connector면 프로젝트 선택 결과를 기준으로 직접 command를 보낸다. not-ready, project mismatch, 연결 실패는 이 옵션으로 숨기지 않는다.

따라서 "버전이 다르면 무조건 거절"은 아니다. 기본은 거절이고, 사용자가 위험을 알고 옵션을 주면 실행한다.

연결 문제를 줄이는 핵심

  • 사용자 입력에서 포트 개념을 제거했다.
  • 명령마다 같은 프로젝트의 최신 listener를 다시 찾는다.
  • main thread command queue와 별개인 /health로 준비 상태를 확인한다.
  • listener가 리로드 중이면 deadline 안에서 다시 resolve하고, 명령 POST 전까지만 재시도한다.
  • 상태 출력과 문서에서는 프로젝트, Unity 버전, connector 버전, PID만 보여준다.

문서 기준: 현재 워킹트리 변경 사항. 내부 HTTP transport 구현의 포트 필드는 실제 연결에 필요하므로 사용자 선택 옵션과 별도로 유지된다.
